# Domo Toolkit v1.3.1 Release Notes (WIP)

## UI Improvements

- Removed pulsing effect on Copy Filters button, as it was distracting more than it added value
- Fixed action buttons growing vertically when their label wrapped onto two lines — buttons now stay at a consistent height and the label wraps inside
- Activity Log Source column (DomoStats path): `USER` chips stay accent and `GROUP` chips are now success-green; any other source type (`SYSTEM`, `ETL`, etc.) hashes to a stable color so each type is visually distinct but consistent across rows
- DataList header title now shows its full value in a HeroUI Tooltip on hover (replacing the previous native `title`-attribute approach), so truncated headings like "Downstream Content of {long dataset name}" are always readable. Subtext deliberately does NOT get a tooltip since every subtext we render is a short, bounded count/status string that wouldn't realistically truncate at typical sidepanel widths
- DataList header title and subtext now take plain strings with inline `**bold**` markdown (instead of JSX) — the shared `parseMarkdownBold` util renders the bold for the visible header while `stripMarkdownBold` flattens it for the title tooltip so the overlay reads as unstyled prose. Migrated every consumer (GetCards, GetPages, GetDatasets, GetViewInputs, Ownership, MigrateDownstreamContent) and extracted the markdown helpers (originally embedded in useStatusBar's `parseDescription`) to `src/utils/markdown.jsx` so status messages, header titles, and subtext all speak the same lightweight markup
- The close (X) button on every DataList view now shows a uniform "Close view" tooltip instead of a per-view label (e.g. "Close Cards View", "Close Ownership View"). The label is hardcoded in DataList, so views no longer pass a `closeLabel` prop
- Context footer related-data tabs (related objects, dataset columns, template approvals, etc.) are now cached in a module-level store keyed by Chrome tab, so switching Chrome tabs and coming back no longer refetches the data while the sidepanel stays open. Each Chrome tab's cache is tied to the object that was detected on it, so navigating to a different object in the same Chrome tab still clears and reloads. Cached entries expire after 300 seconds so data that goes stale (like pending approvals) gets refreshed on the next open, and a tab's cache is dropped when that Chrome tab is closed
- Confirmation dialogs for the secondary delete actions (like "Delete App and All Cards" on a page) now bold the object name and ID in the dialog body, matching the primary delete confirmation. Done by wrapping the names in `**bold**` markers inside each cascade's `confirmText` and rendering through the shared `parseMarkdownBold` util, instead of dropping a raw string into the dialog
- The "Delete App and All Cards" confirmation now shows the page and card counts in parentheses (e.g. "all its pages (4), and all cards on those pages (37)") so the blast radius is clear before confirming. The counts come from a single app admin-summary call (`/api/content/v1/dataapps/{appId}/adminsummary`) made during the dependency check on view load, and those same card IDs are reused for the delete instead of re-walking every page's cards at delete time. Worksheets and any failed summary call degrade gracefully: counts are omitted and the delete falls back to the per-page walk
- Tightened the vertical gap between a disclosed parent row and its children in DataList. HeroUI's default `disclosure__body` padding (8px) stacked on top of the parent heading's and first child's `my-1` margins (4px each), making the parent-to-child gap ~16px, double the ~8px sibling-to-sibling gap (which has no body padding between rows). The expanded children read as disconnected from their header as a result. Dropping just the body's vertical padding to 0 (while keeping its horizontal `px-2`, which supplies the ~8px-per-level indentation) brings the parent-to-child gap in line with the sibling rhythm. Scoped via a `.datalist-root` class so the standalone disclosures elsewhere (API Errors, Dataflow Inspector, etc.) keep HeroUI's default padding, and ordered before the selection-mode `disclosure__body` rule so selection mode still strips the padding entirely for its own explicit `paddingLeft` indentation

## New Features

### Migrate Downstream Content (Datasets)

- New "Migrate Downstream Content" action button on datasets — finds every card, dataset view, and dataflow that uses the in-scope dataset as an input and rewires their input to a new dataset
- Discovery uses Domo's lineage API (no DomoStats setup required) plus the dataset's cards endpoint
- Sidepanel view mirrors the Transfer Ownership UX: every found item is pre-selected, with checkboxes on both type groups and individual items so the user can deselect entire categories or cherry-pick
- Target dataset picker uses async typeahead search with paginated load-more
- Schema-compatibility check runs as soon as a target is picked — surfaces missing/type-mismatched columns inline
- **Column remapper for incompatible schemas**: when the target's schema doesn't fully cover the origin, the modal scans every selected card / dataset view / dataflow for column references and surfaces only the columns that are BOTH actually used by the selected content AND missing from the target. Each surfaced column gets a per-row dropdown of target columns, defaulting to "Leave unmapped" — no auto-suggestions, the user picks each mapping deliberately. The same mapping applies consistently across every piece of content powered by the dataset.
- **Robust column-reference rewriter**: applies the user's column map at three levels — JSON object keys at known column-keyed paths (`columnFormats`, `columnInfo`), plain string values at known column-bearing fields (`column`, `leftColumn`, `groupBy`, etc., plus column-list arrays), and backtick-wrapped column refs inside expression strings (formulas, SQL clauses, `formattedExpression`). Card definitions that get column rewrites are PUT through the full card-update endpoint instead of the lightweight datasource-swap shortcut.
- Strong inline warnings throughout the schema-mismatch flow — best-practice text reminding the user to align schemas before migration, and that broken column references can cause cards to render blank, dataflows to fail, and views to error
- Per-type progress reported live in the sidepanel rows; failures expose the underlying error message
- Dataset-view input-swap recurses through SQL `selectBody` (joins, set operations), updates column `referenceDataSourceId` references, rewrites `formattedExpression` mappings, then does a final string sweep — the same hardened logic from the original CLI tool
- Drill cards on each parent card are also migrated when their content uses the in-scope dataset: discovered via the bulk `parts=drillPath,drillPathURNs` endpoint, fetched with their own datasource metadata, and rewired alongside the parent so drill-down paths don't break after the swap
- Downstream dataflows now show their real names instead of `Dataflow {id}` — the lineage endpoint doesn't return names, so we hydrate them in parallel via `getDataflowDetail` after the lineage fetch
- Parent group checkboxes (Cards / Dataset Views / Dataflows) in selection mode now show the indeterminate dash icon when only some of their children are selected, instead of reading as unchecked. Implemented in `DataList` as a general affordance so any future view with parent-and-child selectable rows gets the same behavior; OwnershipView's existing "Select all" toolbar checkbox already follows the same pattern
- When a dataset has no downstream content (all three lineage fetches settle with zero cards, dataset views, and dataflows), the view fires a "Nothing to migrate" warning toast and returns to the default view instead of painting an empty list. Skips the bail when any fetch errored so the failure stays visible and the user can retry via refresh

### Transfer Ownership (per-item selection)

- Transfer Ownership is no longer all-or-nothing per object type. In selection mode each leaf row (individual card, dataset, dataflow, …) now gets its own checkbox so users can transfer a subset within a type — e.g. move 3 specific dashboards instead of every dashboard the source user owns
- Parent type rows act as cascading toggles: clicking a type checks/unchecks all its leaves, deselecting any leaf flips the parent into indeterminate, and the toolbar "Select all" reads against the leaf-level totals so it goes indeterminate whenever any single object is deselected
- Subtext line in selection mode now shows `**N** types, **M** objects selected` driven by the actual leaf selection (was previously the all-or-nothing type count), and the bottom-pinned "Transfer ownership to…" footer button enables as soon as ≥1 individual object is selected (no longer requires a full type)
- Individual object checkboxes now appear pre-checked as soon as their type's fetch resolves, instead of waiting for every type to finish loading. The optimistic-pre-select effect runs incrementally per type — each loaded type idempotently seeds its leaves (if the user hasn't deselected the parent type during loading) or prunes itself (if it ended up non-eligible). Net effect: a user who launches the Transfer Ownership view sees per-leaf checkboxes light up cards/datasets/dataflows/etc. as their data streams in, instead of every leaf flipping checked together at the slowest fetch's deadline
- The Select-all toolbar checkbox no longer flashes between indeterminate and checked while types are loading. During the loading phase it's pinned to a stable "checked + disabled" visual so each newly-arriving type doesn't briefly outpace the leaf-hydration tick; once everything is fully loaded the toolbar resumes its normal three-state evaluation (unchecked / indeterminate / checked) against the per-leaf selection
- Virtual parent rows (group headers like "Cards", "Dataset Views", "Dataflows", "Pages", "Datasets", etc.) now show the matching ObjectTypeIcon in front of the label — matching the icon already displayed on every leaf inside the group. Applied generically in DataList by routing virtual-parent label rendering through the same `labelInner` helper that regular items use; consumers (OwnershipView, MigrateDownstreamContentView) populate `typeId` on each parent DataListItem from their existing `TYPE_KEY_TO_DOMO_TYPE` map. Synthetic group rows without a typeId (e.g. `REPORT_BUILDER_group`) render unchanged because ObjectTypeIcon returns null when typeId is unmapped
- Tasks now nest under their parent project inside the "Projects & Tasks" row instead of rendering as a flat list. The tree shape mirrors Domo's data model: virtual parent → projects → tasks. Project checkboxes cascade to their task children (selecting a project auto-checks every task underneath); deselecting any single task flips the project checkbox to indeterminate but leaves the project's own selection intact so the user keeps fine-grained control. The `[Project]` / `[Task]` label prefixes are dropped — the tree hierarchy plus the Project icon on every row conveys the role visually. Orphan tasks (whose parent project isn't in the owned list — defensive only, shouldn't happen given how `getOwnedProjectsAndTasks` fetches tasks per-owned-project) render alongside projects as a fallback so they stay selectable
- Fixed project display names that were rendering as numeric IDs in the Transfer Ownership flow. The Domo project API returns the display name in `projectName` (matching the `pathToName: 'projectName'` on the PROJECT object type), but `getOwnedProjectsAndTasks` was reading `project.name` and silently falling back to the id-as-string when it was undefined. Pre-existing bug — the old `[Project] 12345` label format camouflaged it; the new flat label exposed it. Service now reads `projectName` first, with `name` and the id-as-string as fallbacks
- DataList tooltips on plain (non-Link, non-virtual-parent) rows now show the canonical `originalId` when present, instead of the React-key id. Matters for views like the Projects & Tasks tree where the React id is namespaced (`projectsAndTasks:task-12345`) to keep CheckboxGroup values unique — the tooltip used to read out that whole composite string. Now it just shows the underlying task or project id, matching the long-standing behavior of the Copy-ID action
- Nested DataList rows in selection mode now indent by 16px per nesting level so children read as descendants of their parent instead of as flush-left siblings. Most visible in OwnershipView's Projects & Tasks tree (tasks shift right under their project), plus Transfer Ownership and Migrate Downstream Content. The indent is deliberately scoped to selection mode: default (non-selection) views already pick up ~8px of natural indent per level from each level's own `disclosure__body` padding, and HeroUI strips that body padding only in selection mode, so the explicit indent fills exactly the gap that stripping leaves behind without double-padding the default views (a brief unconditional version over-indented normal views like DataFlow DataSets before this scoping landed). The `depth` prop is threaded recursively from `DataList` to `DataListItem` and drives an inline `paddingLeft` on the row's outer container (`Disclosure.Heading` for parents, the flat wrapper div for leaves), applied only to the row's outer element so it doesn't compound with the children's own depth-based padding
- Project and Task rows in OwnershipView's Projects & Tasks tree are now clickable links to their actual Domo pages. Projects open at `/project/{id}` (the canonical PROJECT urlPath); tasks open at `/project?taskId={id}` (the PROJECT\_TASK urlPath). Earlier the rows rendered as plain text because the leaf-builder hardcoded `url: null` — corrected by constructing URLs via `DomoObject` against the appropriate subtype, mirroring how every other transfer type (datasets, cards, dataflows…) already links to its native Domo page
- `transferAllOwnership` in `services/transferOwnership.js` gained an optional `enabledItemIds: Map<typeKey, Set<itemId>>` parameter that post-filters the listed `owned` array per type before the transfer call. Preserves all existing callers (filter is opt-in); works correctly for types with a dedicated `getOwnedForTransfer` (filter applies to the re-fetched canonical list) and for `projectsAndTasks` (filters projects/tasks separately by `project-<id>` / `task-<id>` composites)
- `TransferOwnershipModal` no longer derives object counts from a `selectedTypeKeys` Set + `results` map. The parent now hands it pre-computed `selectedTypeCount` / `selectedObjectCount` numbers, so the modal stays leaf-id agnostic

### Supported Types

- Added DataSet to related objects for Approval Templates.
- Added an Approvals tab to the related objects for Approval Templates. It lazily fetches every active approval request derived from the template when the tab is opened, paging through Domo's cursor-based workflow search so the full set is returned instead of just the first 30. Each row carries the richer fields the Approval Center UI reads (status, last-modified time, version, current pending approver, and submitter) so the request is one click away with context, not just an id and title.
- AppDB Collection now auto-resolves its parent DataStore id from the collection details response (`datastoreId`), so the "DataStore" related-object tab populates without needing the parent id to be pre-supplied at detection time.
- New "Sync Datastore" action button on AppDB collections. Posts to `/api/datastores/v1/export/{datastoreId}` (no body) using the collection's parent datastore id, kicking off the same manual sync Domo's UI fires. Surfaces a toast for in-flight, success, and failure states. AppDB-only: Sync's source file (`Sync.jsx`) no longer covers the Code Engine flow since that flow was never really a sync. The button is gated on the collection's `details.syncEnabled` flag, so collections that haven't had sync turned on (typically because they don't yet have a schema) don't show an action that would no-op against the export endpoint. Using "Apply Schema and Sync" in the Generate Schema flow flips `syncEnabled` on as part of the same PUT, so a one-click path from a freshly-generated schema to a running sync still works.
- Renamed the existing Code Engine "Sync JSDoc to Package" button and its view to "Generate Definition from JSDoc". The action does not sync, it generates the package's function/version manifest from the JSDoc in the IDE source, then writes it. The new label matches what the action actually does, and the source moves to the same `Generate.jsx` file that hosts the new AppDB "Generate Schema" button (both are generators that infer a definition from source data and then optionally write it). View file renamed to `GeneratePackageDefinitionFromJSDocView.jsx`; sidepanel routing key renamed from `syncJSDocFromSource` to `generatePackageDefinitionFromJSDoc`. The button is also now gated to JavaScript packages only (and packages with no language set, which the toolkit treats as JavaScript by default); Python packages no longer surface the action since JSDoc is JavaScript-specific. Detection reads `metadata.details.language` for `CODEENGINE_PACKAGE` and falls back to `metadata.parent.details.language` for `CODEENGINE_PACKAGE_VERSION`, since the version response doesn't carry the package-level language field.
- New "Generate Schema" action button on AppDB collections. Queries the 100 most recent documents in the collection (`POST /api/datastores/v2/collections/{collectionId}/documents/query?limit=100&offset=0&orderby=createdOn+descending`), unions every property key seen across those documents (so sparse keys present in only some docs still produce a column), and infers a column type per key: LONG for whole numbers, DOUBLE for decimals, DATETIME for ISO timestamps, DATE for plain YYYY-MM-DD strings, STRING for everything else (booleans, arrays, objects, mixed, all-null). The inferred list opens in a sidepanel view where every row is editable: rename, retype (STRING, LONG, DOUBLE, DECIMAL, DATE, DATETIME), remove, or add columns. Saves the result via `PUT /api/datastores/v1/collections/{collectionId}` with a `{id, schema: {columns}}` body. Footer offers two paths: **Apply Schema** saves alone; **Apply Schema and Sync** chains three calls in order: the schema PUT, a second PUT to flip `syncEnabled: true` at the collection level (sent as its own request because Domo's collection endpoint does not honor `syncEnabled` when bundled with a schema body), then the existing Sync Datastore export so a fresh DataSet with the new columns drops out of one click. Validation blocks save if any column name is empty, any two columns share a name, or the list is empty. When the collection has zero documents, the view bails with a warning toast instead of opening empty.

### Delete Approval Template (related dataset + combined delete)

- The Delete view now runs a dependency check for approval templates. Under "Other dependencies" it surfaces the template's related dataset by name with a badge showing how many objects sit downstream of it (so "(0 dependencies)" signals it is safe to delete), plus a tally of the template's existing approval requests as a single "Approvals (N requests)" row rather than listing each one.
- Added a second delete option, "Delete Template and DataSet", next to the existing "Delete Template" button. This mirrors the two-option pattern data app views already use ("Delete Page and All Cards" vs "Delete app and all cards"): it deletes the template, then its backing dataset, in one pass.
- The combined option is blocked (button disabled, with a tooltip explaining why) when that downstream count is non-zero, so you can't orphan dependents. The plain "Delete Template" button stays enabled regardless.
- The related dataset's id is read from the template details already enriched at detection (no extra fetch); its name is resolved via dataset search, its dependents via Domo's precomputed impact-count endpoint (summing the downstream dataflows, datasets, cards, and alerts it rolls up), and the approval count via a lightweight workflow-search total that falls back to a full count if an instance does not expose it.
- The Delete view's cascade-button mechanism was generalized so each secondary action supplies its own label, tooltip, confirmation copy, run logic, and optional blocked state (data app and worksheet deletes keep their existing behavior). The primary-delete confirmation now counts only the dependencies that delete actually removes, so a template's confirmation no longer implies the related dataset will be touched.

## Bug Fixes

### Update Owner: user search no longer clears the selected owner on blur

- Fixed the Update Owner (Alert/Workflow) user picker dropping the selected user's name from the search box the moment you clicked out (onto Save or elsewhere), reverting to the empty/required state and blocking submit until you re-searched for the same user and clicked out a second time. Root cause: the modal controlled the ComboBox's `inputValue` but left its `selectedKey` uncontrolled. React Aria's blur-commit (`useComboBoxState.commitSelection`) handles a half-controlled ComboBox by resetting the visible input to the selected item's text looked up in the *current* list, and picking a user refetches page 0 of all users (so the just-picked user is usually no longer in that list), leaving no text to restore. The native `required` on the now-empty input then blocked Save, while the underlying selection survived (which is why re-searching showed the checkmark and a second blur recovered)
- Fix mirrors the already-working Transfer Ownership picker: UpdateOwner now fully controls the ComboBox via a `selectedOwnerId` state (`selectedKey` + `onSelectionChange`) and submits from that state instead of harvesting the form, which routes blur through the safe commit path that leaves the input untouched. Removed the dead form plumbing (`name`, `formValue`, `form`, `defaultInputValue`) and a stray `console.log`
- Also gated the Save button on having a real selection (matching Transfer Ownership) and reset the picked owner each time the modal opens so a stale pick can't leak into the next session

### Duplicate User: scoped sharing, itemized preview, audit-log download

- Fixed Duplicate User over-sharing where the new user ended up with explicit shares on cards and pages the source user only saw indirectly. Root cause: the previous flow fetched from `/api/content/v1/access/users/{id}/cards|pages`, which conflates direct individual grants with content reachable through group membership, PDP rules, dataset-derived visibility, org-wide content, and Workspace membership. Every visibility path was then re-granted as a direct share on the new user, so the per-resource "Shared with" panels filled up with the new user even when the source user had never been there as a direct grantee. The new user's effective access is unchanged because step 5 still copies group memberships (and inherited Workspace access flows from those groups). Only the *direct-share* list per resource gets tidied up
- Replaced the access-endpoint read with a fresh derivation in `src/services/userIndividualShares.js`: pulls the combined `/api/content/v1/access/users/{id}` response, keeps only `accessRecords` where `groupId` is null and `userId` matches the target, then cross-checks each surviving candidate against the Workspaces API (`/api/nav/v1/workspaces/entity/{type}/{id}` and the embedded `members` array per workspace) to drop anything the user reaches via Workspace membership (their own or any of their groups'). Per-call memoization on both the entity-lookup and the per-workspace member fetch so a user with hundreds of access records doesn't fan out into thousands of round trips
- Handles the page-vs-data-app ambiguity in `accessRecords`: a `pageId` could be a dashboard page OR a data app view, and Workspaces tracks them under different entity types (`page` vs `dataApp`). The new pipeline tries `entity/page/{pageId}` first; on empty response it looks up the page's parent app via `/api/content/v1/pages/summary` (the same `dataAppId` field the rest of the codebase already uses for App Studio pages) and retries as `entity/dataApp/{parentAppId}`. Regular dashboard pages with no Workspace fall through correctly; data app views inside a Workspace get correctly classified and filtered. Custom (pro-code) apps from the `domoappId` field skip the Workspace check because Workspaces doesn't surface them by a known entity type, so they survive the filter and rely on the operator's preview checkboxes
- Fails open on Workspaces lookup errors: if the entity or members endpoint returns a non-OK status for one candidate, the candidate survives the filter with a console warning instead of being silently dropped. The operator's per-item checkboxes are the safety net. Failing closed would silently break duplications when Workspaces APIs hiccup, which is worse than over-including a few items
- Preview rewritten to show every individually-shared card, page, and custom app by name with per-item checkboxes (defaults to all-selected). Aggregate sections (Role, Profile, Locale, Group memberships) still render compactly with disclosure expansion for long lists. The three selectable sections render through `DataList` with `selectionMode={true}`, virtualization, and a "Select all / Deselect all" toolbar so a 500-card list stays usable. Inline italic note under each selectable section reminds operators that deselection only suppresses the *direct-share grant*, not group-inherited access, to head off the misconception that deselection locks the new user out of the resource entirely
- Custom apps surface in the preview but the share step is intentionally a no-op for now (Domo's custom-app share endpoint is `shareCustomAppDesign(designId, …)`, which requires a separate `designId` lookup not yet built into this flow). Custom apps that were checked appear in the audit log with `Status: SKIPPED` and `Notes: Manual sharing required` so operators have an explicit record to follow up on
- Auto-downloads an Excel audit log (`duplicated-user_YYYYMMDD_HHMMSS.xlsx`) at the end of every duplication with one row per attempted item across all steps: USER (the create itself), one PROFILE\_FIELD row per copied field, LOCALE, one GROUP row per group added, one CARD/PAGE row per attempted share (status `SHARED` / `FAILED` with the error message in Notes), and one CUSTOM\_APP row per checked custom app. Each row carries Source User ID/Name and New User ID/Name columns for cross-referencing. Built via the existing `buildExcelBlob` + `downloadBlob` plumbing in `src/utils/exportData.js`, same as the activity log export
- Updated terminology throughout: the action button tooltip now reads `Clone this user's role, profile, groups, and individually-shared content`, the two share step labels now read `Share individually-shared cards` and `Share individually-shared pages` (plus a new `Share individually-shared apps` step), and the post-completion toast mentions the audit log was downloaded. Replaces the older "accessible cards/pages" phrasing that hinted at the broader, group-inclusive read
- Deleted the dead helpers `getUserAccessibleCards` (`src/services/cards.js`) and `getUserAccessiblePages` (`src/services/pages.js`). Both were used only by the duplicate flow's previous implementation; grep confirmed no other consumers

### Sync JSDoc to Package: regenerate the `module.exports` block so synced functions resolve in Workflows

- Fixed Sync JSDoc creating package versions whose functions Domo Workflows reported as nonexistent at run time (`Function ... not found in package, is it private?`). Domo's Code Engine IDE hides a trailing `module.exports = {...}` block: it strips the block when loading a version into the editor and regenerates it from the parsed function list on save. The runtime relies on that block to expose functions. Because we read the live editor (block already stripped) and POSTed the source verbatim, the saved version had no exports and every function was unresolvable. Pinpointed with a control test: a manual Domo save of the identical code worked while our sync failed, and the only code difference was the ~540-character export block at the end
- New `appendModuleExports` reattaches the block before POSTing, regenerated from the live editor's function list in declaration order and matching Domo's exact formatting byte-for-byte (validated against a captured manual save). It strips any existing trailing block first so repeated syncs never stack duplicates, and it is applied only to the POSTed code, not written back into the editor (preserving Domo's hidden-block convention)
- `getCodeEngineEditorSource` now walks the live CodeMirror 6 Lezer tree while reading the source and returns the ordered `functionNames` (for the export block) plus `editorStartIndices` (below). Sync is blocked, with an explanatory message, when that tree can't be read (editor unreachable, or the saved-version API fallback), since a version saved without the export block would reintroduce the bug
- Also corrected `editorStartIndex`, which we previously hardcoded to 0 for every function. Domo binds each manifest function to its code definition by this parse-tree node index; we now emit the real value from the same Lezer walk (verified 27/27 against a manual save). This was not the cause of the Workflow failure, but it makes our saved version match Domo's byte-for-byte

### Update Code Engine Versions: reconcile changed inputs/outputs instead of breaking the tile

- Bumping a Code Engine function version used to write only the new version number onto the tile, leaving its input bindings and output wiring untouched. When the new version changed the function's contract (a new output, a renamed/added/removed input, a type change) the tile silently went stale: new outputs never appeared, and renamed inputs lost their binding. The view now diffs each changed action's old vs new function contract (`getCodeEnginePackageVersion` per version, cached by `packageId@version`, fetched only for actions that actually change) and reconciles the tile so the workflow keeps working
- Reconciliation is inline and only shows up when a contract actually changed, so the common "bump to latest, nothing else changed" case stays one click. An affected action gets a collapsible panel tagged Auto (handled for you) or Review (needs a decision)
- Renamed inputs and outputs are re-pointed to the **same** workflow variable automatically. Because Domo binds tile params to shared variables in `dataList` (not tile-to-tile), keeping the variable means every downstream reader keeps working with no further edits
- New outputs are added to the tile and, by default, mapped to a freshly created variable so they are immediately usable downstream (opt out per output with a checkbox). New optional inputs are added silently; new required inputs are flagged to set in Domo after updating
- Removed/renamed inputs that carried a binding get a dropdown to map that binding onto a new input or drop it. Type changes surface a warning naming the bound variable and every other tile that shares it, with an opt-in to update the variable's type. Removed outputs whose variable still feeds downstream tiles warn which tiles would break
- The only hard block is a function that no longer exists in the target version: that one action is skipped (with a clear warning) while the rest still apply. Everything else warns but lets the user proceed

## Refactoring

- Removed the `src/utils/jsdocToPackage/index.js` barrel, per the no-barrel-files convention. `parseSourceToManifest` moved into its own file (`parseSourceToManifest.js`), and the two consumers (`ceContractDiff.js`, `SyncJSDocFromSourceView.jsx`) now import directly from `mergeManifest`, `moduleExports`, and `parseSourceToManifest`.
- Finished the barrel sweep: removed the last four barrels (`src/data/index.js` and `src/lineage/services|hooks|components/index.js`) and rewired their consumers (`background.js`, `useReleaseNotification`, `ReleaseNotes`, `Lineage`, `useLineageCache`, `DataflowInspector`) to import directly from source. The codebase now has no barrel or re-export `index` files.
