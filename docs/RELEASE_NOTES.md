# Domo Toolkit v1.3.1 Release Notes (WIP)

## UI Improvements

- Removed pulsing effect on Copy Filters button, as it was distracting more than it added value
- Fixed action buttons growing vertically when their label wrapped onto two lines — buttons now stay at a consistent height and the label wraps inside
- Activity Log Source column (DomoStats path): `USER` chips stay accent and `GROUP` chips are now success-green; any other source type (`SYSTEM`, `ETL`, etc.) hashes to a stable color so each type is visually distinct but consistent across rows
- DataList header title now shows its full value in a HeroUI Tooltip on hover (replacing the previous native `title`-attribute approach), so truncated headings like "Downstream Content of {long dataset name}" are always readable. Subtext deliberately does NOT get a tooltip since every subtext we render is a short, bounded count/status string that wouldn't realistically truncate at typical sidepanel widths
- DataList header title and subtext now take plain strings with inline `**bold**` markdown (instead of JSX) — the shared `parseMarkdownBold` util renders the bold for the visible header while `stripMarkdownBold` flattens it for the title tooltip so the overlay reads as unstyled prose. Migrated every consumer (GetCards, GetPages, GetDatasets, GetViewInputs, Ownership, MigrateDownstreamContent) and extracted the markdown helpers (originally embedded in useStatusBar's `parseDescription`) to `src/utils/markdown.jsx` so status messages, header titles, and subtext all speak the same lightweight markup

## New Features

### Migrate Downstream Content (Datasets)

- New "Migrate Downstream Content" action button on datasets — finds every card, dataset view, and dataflow that uses the in-scope dataset as an input and rewires their input to a new dataset
- Discovery uses Domo's lineage API (no DomoStats setup required) plus the dataset's cards endpoint
- Sidepanel view mirrors the Transfer Ownership UX: every found item is pre-selected, with checkboxes on both type groups and individual items so the user can deselect entire categories or cherry-pick
- Target dataset picker uses async typeahead search with paginated load-more
- Schema-compatibility check runs as soon as a target is picked — surfaces missing/type-mismatched columns inline
- **Column remapper for incompatible schemas**: when the target's schema doesn't fully cover the origin, the modal scans every selected card / dataset view / dataflow for column references and surfaces only the columns that are BOTH actually used by the selected content AND missing from the target. Each surfaced column gets a per-row dropdown of target columns, defaulting to "Leave unmapped" — no auto-suggestions, the user picks each mapping deliberately. The same mapping applies consistently across every piece of content powered by the dataset.
- **Robust column-reference rewriter**: applies the user's column map at three levels — JSON object keys at known column-keyed paths (`columnFormats`, `columnInfo`), plain string values at known column-bearing fields (`column`, `leftColumn`, `groupBy`, etc., plus column-list arrays), and backtick-wrapped column refs inside expression strings (formulas, SQL clauses, `formattedExpression`). Card definitions that get column rewrites are PUT through the full card-update endpoint instead of the lightweight datasource-swap shortcut.
- Strong inline warnings throughout the schema-mismatch flow — best-practice text reminding the user to align schemas before migration, and that broken column references can cause cards to render blank, dataflows to fail, and views to error
- Per-type progress reported live in the sidepanel rows; failures expose the underlying error message
- Dataset-view input-swap recurses through SQL `selectBody` (joins, set operations), updates column `referenceDataSourceId` references, rewrites `formattedExpression` mappings, then does a final string sweep — the same hardened logic from the original CLI tool
- Drill cards on each parent card are also migrated when their content uses the in-scope dataset: discovered via the bulk `parts=drillPath,drillPathURNs` endpoint, fetched with their own datasource metadata, and rewired alongside the parent so drill-down paths don't break after the swap
- Downstream dataflows now show their real names instead of `Dataflow {id}` — the lineage endpoint doesn't return names, so we hydrate them in parallel via `getDataflowDetail` after the lineage fetch
- Parent group checkboxes (Cards / Dataset Views / Dataflows) in selection mode now show the indeterminate dash icon when only some of their children are selected, instead of reading as unchecked. Implemented in `DataList` as a general affordance so any future view with parent-and-child selectable rows gets the same behavior; OwnershipView's existing "Select all" toolbar checkbox already follows the same pattern

### Transfer Ownership (per-item selection)

- Transfer Ownership is no longer all-or-nothing per object type. In selection mode each leaf row (individual card, dataset, dataflow, …) now gets its own checkbox so users can transfer a subset within a type — e.g. move 3 specific dashboards instead of every dashboard the source user owns
- Parent type rows act as cascading toggles: clicking a type checks/unchecks all its leaves, deselecting any leaf flips the parent into indeterminate, and the toolbar "Select all" reads against the leaf-level totals so it goes indeterminate whenever any single object is deselected
- Subtext line in selection mode now shows `**N** types, **M** objects selected` driven by the actual leaf selection (was previously the all-or-nothing type count), and the bottom-pinned "Transfer ownership to…" footer button enables as soon as ≥1 individual object is selected (no longer requires a full type)
- Individual object checkboxes now appear pre-checked as soon as their type's fetch resolves, instead of waiting for every type to finish loading. The optimistic-pre-select effect runs incrementally per type — each loaded type idempotently seeds its leaves (if the user hasn't deselected the parent type during loading) or prunes itself (if it ended up non-eligible). Net effect: a user who launches the Transfer Ownership view sees per-leaf checkboxes light up cards/datasets/dataflows/etc. as their data streams in, instead of every leaf flipping checked together at the slowest fetch's deadline
- The Select-all toolbar checkbox no longer flashes between indeterminate and checked while types are loading. During the loading phase it's pinned to a stable "checked + disabled" visual so each newly-arriving type doesn't briefly outpace the leaf-hydration tick; once everything is fully loaded the toolbar resumes its normal three-state evaluation (unchecked / indeterminate / checked) against the per-leaf selection
- Virtual parent rows (group headers like "Cards", "Dataset Views", "Dataflows", "Pages", "Datasets", etc.) now show the matching ObjectTypeIcon in front of the label — matching the icon already displayed on every leaf inside the group. Applied generically in DataList by routing virtual-parent label rendering through the same `labelInner` helper that regular items use; consumers (OwnershipView, MigrateDownstreamContentView) populate `typeId` on each parent DataListItem from their existing `TYPE_KEY_TO_DOMO_TYPE` map. Synthetic group rows without a typeId (e.g. `REPORT_BUILDER_group`) render unchanged because ObjectTypeIcon returns null when typeId is unmapped
- Tasks now nest under their parent project inside the "Projects & Tasks" row instead of rendering as a flat list. The tree shape mirrors Domo's data model: virtual parent → projects → tasks. Project checkboxes cascade to their task children (selecting a project auto-checks every task underneath); deselecting any single task flips the project checkbox to indeterminate but leaves the project's own selection intact so the user keeps fine-grained control. The `[Project]` / `[Task]` label prefixes are dropped — the tree hierarchy plus the Project icon on every row conveys the role visually. Orphan tasks (whose parent project isn't in the owned list — defensive only, shouldn't happen given how `getOwnedProjectsAndTasks` fetches tasks per-owned-project) render alongside projects as a fallback so they stay selectable
- Fixed project display names that were rendering as numeric IDs in the Transfer Ownership flow. The Domo project API returns the display name in `projectName` (matching the `pathToName: 'projectName'` on the PROJECT object type), but `getOwnedProjectsAndTasks` was reading `project.name` and silently falling back to the id-as-string when it was undefined. Pre-existing bug — the old `[Project] 12345` label format camouflaged it; the new flat label exposed it. Service now reads `projectName` first, with `name` and the id-as-string as fallbacks
- DataList tooltips on plain (non-Link, non-virtual-parent) rows now show the canonical `originalId` when present, instead of the React-key id. Matters for views like the Projects & Tasks tree where the React id is namespaced (`projectsAndTasks:task-12345`) to keep CheckboxGroup values unique — the tooltip used to read out that whole composite string. Now it just shows the underlying task or project id, matching the long-standing behavior of the Copy-ID action
- Nested DataList rows now indent by 16px per nesting level so children read as descendants of their parent instead of as flush-left siblings. Most visible in OwnershipView's Projects & Tasks tree (tasks shift right under their project) and in any future view with multi-level grouping. The `depth` prop was already being threaded recursively from `DataList` → `DataListItem`; it now drives an inline `paddingLeft` on the row's outer container (`Disclosure.Heading` for parents, the flat wrapper div for leaves) — applied only to the row's outer element so it doesn't compound with the children's own depth-based padding
- Project and Task rows in OwnershipView's Projects & Tasks tree are now clickable links to their actual Domo pages. Projects open at `/project/{id}` (the canonical PROJECT urlPath); tasks open at `/project?taskId={id}` (the PROJECT_TASK urlPath). Earlier the rows rendered as plain text because the leaf-builder hardcoded `url: null` — corrected by constructing URLs via `DomoObject` against the appropriate subtype, mirroring how every other transfer type (datasets, cards, dataflows…) already links to its native Domo page
- `transferAllOwnership` in `services/transferOwnership.js` gained an optional `enabledItemIds: Map<typeKey, Set<itemId>>` parameter that post-filters the listed `owned` array per type before the transfer call. Preserves all existing callers (filter is opt-in); works correctly for types with a dedicated `getOwnedForTransfer` (filter applies to the re-fetched canonical list) and for `projectsAndTasks` (filters projects/tasks separately by `project-<id>` / `task-<id>` composites)
- `TransferOwnershipModal` no longer derives object counts from a `selectedTypeKeys` Set + `results` map. The parent now hands it pre-computed `selectedTypeCount` / `selectedObjectCount` numbers, so the modal stays leaf-id agnostic
